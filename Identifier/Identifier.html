Valid Identifier Or Not

Identifier refers to name given to entities such as variables, functions, structures etc.

Identifiers must be unique. They are created to give a unique name to an entity to identify it during the execution of the program.
Also remember, identifier names must be different from keywords. You cannot use int as an identifier because int is a keyword.
Rules for Naming identifier
  1. A valid identifier can have letters (both uppercase and lowercase letters), digits and underscores.
  2. The first letter of an identifier should be either a letter or an underscore.
  3. You cannot use keywords like int, while etc. as identifiers.

Process

Detection of the programming language’s reserved words (aka keywords) is typically baked into the lexical analyzer (aka scanner, tokenizer, etc.) portion of the compiler. The lexical analyzer breaks the source code into tokens, and when it sees a keyword (or reserved word), it returns a token associated with that specific keyword to the parser. (In some lexical analyzers, keywords are initially internally recognized as identifiers, and are then looked up in a table of keywords, before returning the appropriate keyword token. But this approach is not typical.)
Detection of identifiers is similarly built into the lexical analyzer. The definition of a programming language’s valid identifiers includes a set of rules (e.g., it must begin with a letter or underscore, must be followed by a sequence of letters/digits/underscores, must not exceed some maximum length, etc.). In some lexical analyzers, new identifiers are entered into the symbol table, and subsequent occurrences of the identifier return the same identifier information to the parser, along with the identifier token. But some lexical analyzers leave symbol table management to the parser.
Inside the lexical analyzer, recognition of tokens is typically accomplished with an FSA (finite state automaton). An FSA is sometimes constructed using regular-expression-like statements in compiler development tool like Lex or Flex. Effectively, an FSA is a state machine, where state transitions occur on characters read from the source code by the lexical analyzer.
